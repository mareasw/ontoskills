通过 MCP 工具传递的结构化知识是否真的能帮助 AI 代理比原始 Markdown 文件更好地完成任务？我们进行了一项对照实验来验证。

## 核心问题

像 Claude Code 这样的 AI 编程代理依赖技能文档来完成专业任务——生成 DOCX 文件、处理 PDF、分析金融数据。如今，这些技能以纯 Markdown 文件（`SKILL.md`）的形式提供。代理必须阅读原始文本并自行提取指令、启发式规则和反模式。

**OntoSkills** 采用了不同的方法：技能知识被编译成结构化的 OWL 2 本体，通过 MCP 工具传递。代理使用 `prefetch_knowledge` 一次性加载结构化技能知识——接收带有严重性评级、认知规则和执行计划评估的类型化知识节点。

哪种方法效果更好？

## SkillsBench：确定性代码生成评估

我们使用 [SkillsBench](https://github.com/benchflow-ai/skillsbench) 对两种方法进行了评估，该基准测试衡量代理为真实任务生成可用代码的能力。

### 评估方式

1. 代理接收任务描述和相关的技能文档
2. 生成一个 Python 解决方案脚本
3. 脚本在任务的 **Docker 容器**中运行（通过 podman）
4. **pytest 测试套件**验证输出文件——完全确定性，无需人工判断
5. 分数 = `通过测试数 / 总测试数`

这不是 LLM 评审。评估完全确定且可重复。

### 实验设置

| 参数 | 值 |
|------|-----|
| 代理 | Claude Code CLI（`--print --bare` 模式）|
| 模型 | glm-5.1（通过 API 代理）|
| 任务数 | 24（seed=7），来自 70+ 个可用任务池 |
| 尝试次数 | 每任务 3 次（多轮 Docker 执行反馈）|
| 评分 | Docker + pytest CTRF 报告，3 次尝试中取最佳 |

### 代理模式

**传统模式** — 技能文档以 SKILL.md 文件形式放置在 `.claude/skills/` 中。代理使用 Claude Code 的原生文件读取功能发现和加载技能——与生产环境中的工作方式完全一致。

**OntoSkills MCP 模式** — 技能编译为 OWL 2 本体，通过 OntoMCP 提供。代理通过 MCP 工具（`prefetch_knowledge`、`search`、`get_skill_context`）发现技能。`ontomcp-driver` 技能教导代理查询本体的最佳工作流程。

两种模式使用相同的 Claude Code 代理、相同的模型、相同的提示。唯一的区别是**技能知识的传递方式**。

## 结果

<BenchmarkApp />

### 主要发现

- **传统模式通过率更高**：50%（12/24）vs 37.5%（9/24）——原始 Markdown 配合多轮反馈非常有效
- **OntoSkills 更高效**：输入 token 减少 11%，输出 token 减少 7%——结构化知识更紧凑
- **OntoSkills 成本更低**：总计 $10.38 vs $11.53（-10%），因为 token 使用量更低
- **两种模式都受益于多轮反馈**：3 次尝试的反馈循环改善了两种模式的结果（传统：41.7%→50%，MCP：25%→37.5%）

### 诚实分析

传统模式更高的通过率源于代理从一开始就拥有完整的 SKILL.md 内容——它可以一次性读取全部技能文档并跨文件交叉引用。当添加多轮反馈（将 Docker 测试结果反馈给代理进行修复）时，传统代理拥有足够的上下文来有效地自我纠正。

OntoSkills 的结构化知识传递在**token 效率**方面表现出色——代理接收经过优先级排序的类型化知识节点，而非原始文本。然而，对于此基准测试的代码生成任务，拥有完整的 SKILL.md 内容比拥有结构化但可能被过滤的知识更有用。

差距从初始运行显著缩小（传统领先 +16.7pp → +12.5pp），因为 OntoSkills 在相对意义上从多轮反馈中获益更多（+50% 改善 vs 传统的 +20%）。

### 基础设施故障

1 个任务因与技能质量无关的**基础设施问题**而在两种模式下都失败了：
- `flink-query` — Java/Flink 容器没有 Python 解释器；基准测试始终运行 `python3 /tmp/agent_solution.py`

## 为什么结构化知识仍然重要

虽然传统模式在代码生成任务的原始通过率上领先，但 OntoSkills 更高效地传递知识：

- **带有严重性评级的类型化节点**让代理优先处理关键规则
- **带有原因的反模式**解释*为什么*要避免特定方法
- **执行计划评估**在编码前捕获错误
- **11% 更少的输入 token** 在大规模应用时转化为显著的成本节省

token 效率优势随着技能目录的增大而增长——当代理需要导航数百个技能而非少数几个时，通过 MCP 进行的结构化传递变得越来越有价值。

## 局限性

- **单一模型**：所有结果使用 glm-5.1 通过 API 代理。其他模型的表现可能不同。
- **单一基准**：SkillsBench 测试代码生成。其他基准（GAIA 问答、SWE-bench 代码库修补）将测试不同能力。
- **种子依赖**：任务选择因种子而异。我们报告 seed=7 以确保可重复性。
- **BM25 节点过滤**：MCP 代理使用 BM25 排序的知识节点，可能会过滤掉某些任务的重要上下文。

## 后续计划

- **改进节点过滤** — 调优 BM25 参数以保留更多相关知识
- **测试驱动提示** — 将测试规范注入提示以提高首次尝试成功率
- **GAIA** 评估（带文件附件的问答）— 需要 HuggingFace 认证
- **SWE-bench** 评估（代码库修补）— 已计划使用更新模型
